## What This Post Covers

The previous post on transformers covered the high-level architecture: embedding layers, attention, and MLP layers working together to predict the next word in a sequence. This post goes deeper — into the machinery that makes that possible. We will build up from scratch, starting with how raw text becomes numbers and ending with the full multi-head attention mechanism that sits at the heart of every modern large language model (LLM).

The concepts here are drawn from *"Build a Large Language Model (From Scratch)"* by Sebastian Raschka. Throughout, we will ground each idea in a concrete running example — the sentence **"The cat sat on the mat"** — and frame explanations around next-word prediction in the style of GPT.

## Tokenization: Turning Text Into Numbers

Neural networks cannot process raw text. Before any learning can happen, text must be converted into numbers. This process is called **tokenization**.

### From Characters to Tokens

A **token** is the basic unit of text that a model operates on. Tokens can be characters, words, or subword chunks — the choice depends on the tokenization strategy. Given a sentence:

```
"The cat sat on the mat"
```

A simple word-level tokenizer splits on spaces and produces:

```
["The", "cat", "sat", "on", "the", "mat"]
```

### Token IDs and the Vocabulary

Once we have tokens, each unique token is assigned an integer ID by building a **vocabulary** — a dictionary mapping every token the model knows to a unique number:

```python
vocabulary = {
    "The":  0,
    "cat":  1,
    "sat":  2,
    "on":   3,
    "the":  4,
    "mat":  5,
    ...
}
```

Our sentence then becomes a sequence of token IDs:

```
[0, 1, 2, 3, 4, 5]
```

This sequence of integers is what gets fed into the model. The vocabulary size matters: GPT-2 has a vocabulary of 50,257 tokens. Larger vocabularies can represent more words precisely but require more memory.

### Byte Pair Encoding (BPE)

Simple word-level tokenization has a critical weakness: any word not seen during training is **out-of-vocabulary (OOV)** and cannot be represented. A completely new word like "ChatGPT" would be unknown to a model trained without it.

**Byte Pair Encoding (BPE)** solves this by operating at the subword level. Instead of treating whole words as tokens, BPE learns to split words into frequently occurring subword chunks. The algorithm works as follows:

1. Start with every individual character as its own token
2. Count all adjacent pairs of tokens in the training corpus
3. Merge the most frequent pair into a single new token
4. Repeat until the vocabulary reaches the desired size

For example, starting from characters:

```
"l o w e r"  →  merge "e r" → "l o w er"  →  merge "lo" → "low er"  →  ...
```

The result is a vocabulary of subword pieces. Common words like "the" become a single token. Rare or new words are split into recognizable pieces:

```
"unhappiness"  →  ["un", "happiness"]
"ChatGPT"      →  ["Chat", "G", "PT"]
```

This means BPE can handle any word, even ones never seen during training, by decomposing them into known subword pieces. GPT-2 and GPT-3 both use BPE.

> **Note:** BPE tokenization is learned from the training corpus, not hand-crafted. The merge rules are a byproduct of training, not human design.

## Word Embeddings: Skip-gram and CBOW

Token IDs are integers, but integers have no meaningful geometry — the fact that "cat" is ID 1 and "mat" is ID 5 says nothing about whether they are related. We need a richer representation.

**Word embeddings** map each token ID to a dense vector of real numbers. Similar words end up with similar vectors — vectors that point in the same direction in high-dimensional space. The similarity between two word vectors can be measured with the **dot product**: a high dot product means the vectors are aligned, indicating semantic similarity.

Two foundational algorithms for learning word embeddings are **Skip-gram** and **CBOW**, both part of the **word2vec** family.

### The Core Idea: Learning from Context

Both algorithms are based on the distributional hypothesis: **words that appear in similar contexts have similar meanings**. "Dog" and "puppy" appear near words like "bark", "leash", and "fetch" — so their embeddings should be similar.

### Skip-gram

Skip-gram takes a **center word** as input and tries to predict the **surrounding context words**.

Given the sentence `"The cat sat on the mat"` with a context window of 2:

```
Center word: "sat"
Target predictions: ["The", "cat", "on", "the"]
```

The model learns: given "sat", what words are likely nearby? Over millions of such examples, the embeddings adjust so that words appearing in similar contexts end up with similar vectors.

### CBOW (Continuous Bag of Words)

CBOW is the reverse: it takes the **surrounding context words** as input and predicts the **center word**.

```
Context input:  ["The", "cat", "on", "the"]
Target output:  "sat"
```

CBOW averages the embeddings of the context words and uses that average to predict the center word. It is generally faster to train than Skip-gram and works better on frequent words. Skip-gram tends to work better on rare words and smaller datasets.

| | Skip-gram | CBOW |
|---|---|---|
| **Input** | Center word | Context words |
| **Predicts** | Context words | Center word |
| **Better for** | Rare words, small data | Frequent words, large data |
| **Speed** | Slower | Faster |

> **Note:** In modern LLMs like GPT, embeddings are not pre-trained with Skip-gram or CBOW — they are learned end-to-end as part of training the full transformer. Skip-gram and CBOW are standalone embedding methods, but understanding them builds intuition for why embedding geometry matters.

## Sliding Window Data Sampling

Before training, we need to decide how to turn a long corpus of text into individual training examples. For next-word prediction, we use a technique called **sliding window sampling**.

Given a long sequence of token IDs — say, the first 8 tokens of a corpus:

```
[2, 5, 9, 3, 1, 7, 4, 8]
```

We define a **context length** (also called the sequence length or block size) — the maximum number of tokens the model sees at once. With a context length of 4 and a stride of 1, the sliding window produces the following input-target pairs:

```
Input:  [2, 5, 9, 3]   →  Target: [5, 9, 3, 1]
Input:  [5, 9, 3, 1]   →  Target: [9, 3, 1, 7]
Input:  [9, 3, 1, 7]   →  Target: [3, 1, 7, 4]
Input:  [3, 1, 7, 4]   →  Target: [1, 7, 4, 8]
```

Each input token predicts the next token — so for input `[2, 5, 9, 3]`, the targets are `[5, 9, 3, 1]`, meaning: given token `2`, predict `5`; given `[2, 5]`, predict `9`; and so on.

The **stride** controls how far the window moves each step. A stride of 1 produces the maximum number of overlapping samples. A larger stride produces fewer samples but reduces redundancy — useful when the training corpus is very large.

## Simple Self-Attention

We now have token embeddings — each token in our sequence is a vector. The next question is: how does each token incorporate information from the other tokens in the sequence? This is what **self-attention** does.

We will build up to the full self-attention mechanism in two steps. First, a simplified version with no trainable weights, to build intuition. Then the full Q, K, V formulation.

### The Goal

Consider the sentence `"The cat sat on the mat"`. The word "sat" needs to be interpreted in context — it is not just a verb in isolation, but an action performed by "cat" in a location near "mat". Self-attention allows each token's vector to be updated by aggregating information from all other tokens, weighted by relevance.

### Step 1 — Attention Scores as Dot Products

For a query token `x_query` (say, the embedding for "sat"), we compute how relevant every other token `xᵢ` is by taking the **dot product**:

```
attention_score(xᵢ, x_query) = xᵢ · x_query
```

A higher dot product means the two vectors point in similar directions — the token is more relevant to the query.

For our 6-token sentence, this gives 6 raw attention scores — one per token.

### Step 2 — Normalize to Get Attention Weights

Raw dot products can be large or negative. We normalize them into proper weights that sum to 1 using the **softmax** function:

```
attention_weight(xᵢ) = softmax(attention_scores)ᵢ
```

The result is a probability distribution over the input tokens — how much attention "sat" should pay to each other word.

```
         The   cat   sat   on    the   mat
weights: 0.05  0.30  0.20  0.15  0.10  0.20
                                  (sums to 1.0)
```

### Step 3 — Compute the Context Vector

Finally, we compute the **context vector** for "sat" by taking a weighted sum of all input vectors:

```
context_vector = Σᵢ attention_weight(xᵢ) · xᵢ
```

This context vector is the new, enriched representation of "sat" — a blend of all input tokens, weighted by their relevance. Words like "cat" (0.30) contribute more to "sat"'s updated meaning than "the" (0.05).

This entire process is repeated for every token in the sequence, producing a new context vector for each one.

## Self-Attention with Queries, Keys, and Values

The simple self-attention above has a limitation: the same input vector is used for everything — for asking "what am I looking for?" and for answering "what do I contain?". In practice, these are different roles and should be handled by different learned projections.

Full self-attention introduces three learned weight matrices:

- **Wq** — projects each input into a **Query** vector: *"what am I looking for?"*
- **Wk** — projects each input into a **Key** vector: *"what do I contain?"*
- **Wv** — projects each input into a **Value** vector: *"what should I pass forward if selected?"*

For each input token embedding `xᵢ`:

```
Qᵢ = xᵢ · Wq
Kᵢ = xᵢ · Wk
Vᵢ = xᵢ · Wv
```

### Computing Attention

The attention score between a query token `q` and every key `Kᵢ` is:

```
score(q, Kᵢ) = Q · Kᵢᵀ
```

These scores are then **scaled** by dividing by √dₖ (where dₖ is the dimension of the key vectors). This prevents dot products from growing very large as dimensionality increases, which would push softmax into regions of near-zero gradient:

```
scaled_score = Q · Kᵢᵀ / √dₖ
```

The scaled scores are passed through softmax to get attention weights, and the output is a weighted sum of the Value vectors:

```
output = Σᵢ softmax(Q · Kᵢᵀ / √dₖ) · Vᵢ
```

Written compactly for all queries at once using matrix notation:

```
Attention(Q, K, V) = softmax(QKᵀ / √dₖ) · V
```

This is the **scaled dot-product attention** formula at the core of every transformer.

The key improvement over simple self-attention is that Wq, Wk, and Wv are **learned parameters**. The model learns what to query for, what to advertise in keys, and what to pass forward in values — all tuned by gradient descent to minimize the loss.

## Causal Attention

Standard self-attention allows every token to attend to every other token — including future tokens. This is fine for tasks like classification or translation (where the full input is available at once), but it breaks next-word prediction.

During training for next-word prediction, the model should not be able to "cheat" by looking at the answer. Given the sequence `["The", "cat", "sat", "on", "the", "mat"]`, when predicting the word after "cat", the model must not see "sat", "on", "the", or "mat".

**Causal attention** (also called masked self-attention) enforces this by **masking** the upper triangle of the attention score matrix — setting those scores to −∞ before the softmax, which causes them to become 0 after softmax:

```
Attention score matrix (before masking):

        The   cat   sat   on   the   mat
The   [ 0.8   0.3   0.5   0.2   0.4   0.1 ]
cat   [ 0.6   0.9   0.7   0.3   0.5   0.2 ]
sat   [ 0.4   0.7   0.8   0.6   0.4   0.3 ]
on    [ 0.2   0.4   0.6   0.9   0.7   0.5 ]
the   [ 0.1   0.3   0.4   0.7   0.8   0.6 ]
mat   [ 0.2   0.4   0.5   0.6   0.7   0.9 ]

After causal masking (upper triangle → −∞ → 0 after softmax):

        The   cat   sat   on   the   mat
The   [ 0.8   —     —     —     —     —   ]
cat   [ 0.6   0.9   —     —     —     —   ]
sat   [ 0.4   0.7   0.8   —     —     —   ]
on    [ 0.2   0.4   0.6   0.9   —     —   ]
the   [ 0.1   0.3   0.4   0.7   0.8   —   ]
mat   [ 0.2   0.4   0.5   0.6   0.7   0.9 ]
```

Each token can only attend to itself and tokens that came before it. This is what makes GPT-style models **autoregressive**: at inference time, they generate one token at a time, always conditioning only on what has been generated so far.

> **Note:** Causal masking is applied during both training and inference. During training it prevents cheating. During inference it is naturally satisfied because future tokens simply do not exist yet.

## Multi-Head Attention

A single self-attention head produces one context vector per token — one view of which tokens are relevant to which. But language has many simultaneous relationships: syntactic (subject-verb agreement), semantic (word meaning), positional (nearby words), and more.

**Multi-head attention** runs several self-attention operations in parallel, each with its own independent Wq, Wk, Wv weight matrices. Each **head** learns to focus on a different type of relationship.

### The Mechanism

Given `h` heads, each head `i` computes its own attention output:

```
headᵢ = Attention(Q·Wqᵢ, K·Wkᵢ, V·Wvᵢ)
```

The outputs of all heads are then **concatenated** and projected through a final learned weight matrix Wo:

```
MultiHead(Q, K, V) = Concat(head₁, head₂, ..., headₕ) · Wo
```

### Why It Matters

Each head operates on a lower-dimensional subspace — if the model dimension is 512 and there are 8 heads, each head works in a 64-dimensional space. This is computationally similar to a single full-dimensional attention operation, but produces richer representations because the heads specialize independently.

> **Note:** The model dimension (`d_model`) is the size of every token's vector throughout the entire transformer. A `d_model` of 512 means every token is represented as 512 numbers at every layer — input, output, and everything in between. This number is a design choice: GPT-3 uses 12,288. Larger dimensions give the model more capacity to represent complex relationships but are more expensive to train.

For example, in a trained GPT model, one head might learn to track subject-verb relationships, another might focus on coreference (connecting "it" to the noun it refers to), and another might attend to positional proximity.

> **Note:** We cannot explicitly tell each head what to focus on — this specialization emerges from training. After training, the attention patterns of individual heads can be visualized and interpreted, but they are not pre-programmed.

## Encoder-Only, Decoder-Only, and Encoder-Decoder Architectures

The transformer architecture is flexible. Depending on the task, different configurations of encoder and decoder blocks are used. Understanding the distinction is essential for understanding why GPT, BERT, and T5 behave so differently despite all being "transformers".

### The Encoder Block

An **encoder** block processes an input sequence and produces a contextual representation of it. It uses **bidirectional self-attention** — every token can attend to every other token, past and future. This makes encoders excellent at understanding and representing text.

### The Decoder Block

A **decoder** block generates an output sequence one token at a time. It uses **causal (masked) self-attention** — each token can only attend to previous tokens. Decoders are designed for generation.

In encoder-decoder models, the decoder also has a **cross-attention** layer that allows it to attend to the encoder's output — connecting understanding of the input to generation of the output.

### Encoder-Only Models

**Architecture:** Stack of encoder blocks only, bidirectional attention.

**Examples:** BERT, RoBERTa

**Best for:** Tasks that require deep understanding of the full input — classification, named entity recognition, question answering (extractive), semantic similarity.

**Why:** Because every token can attend to every other token, encoder-only models build richer representations of meaning. There is no generation step — a classification head is placed on top of the final representations.

```
Input:  "The cat sat on the mat"
         ↓ bidirectional attention (all tokens see all tokens)
Output: [rich contextual representation of each token]
         ↓ classification head
Result: sentiment / label / entity tag
```

### Decoder-Only Models

**Architecture:** Stack of decoder blocks only, causal attention.

**Examples:** GPT-2, GPT-3, GPT-4, LLaMA, Mistral

**Best for:** Text generation, next-word prediction, chat, code generation.

**Why:** Causal attention naturally supports autoregressive generation — predict the next token, append it, predict the next, and so on. The entire pretraining objective (next-word prediction) is perfectly aligned with this architecture.

```
Input:  "The cat sat on"
         ↓ causal attention (each token sees only past tokens)
Output: probability distribution over vocabulary
Result: "the" (most likely next token)
```

### Encoder-Decoder Models

**Architecture:** Encoder stack + Decoder stack, connected by cross-attention.

**Examples:** T5, BART, the original transformer (Vaswani et al. 2017)

**Best for:** Sequence-to-sequence tasks where the input and output are different sequences — translation, summarization, question answering (generative).

**Why:** The encoder reads and understands the full input (bidirectionally). The decoder then generates the output token by token, using cross-attention to condition each generated token on the encoder's representation of the input.

```
Input:   "The cat sat on the mat"  (→ encoder → rich representation)
                                         ↓ cross-attention
Decoder: generates "Le chat s'est assis sur le tapis"  (French translation)
```

> **Note:** Encoder-decoder models have become less common in practice. It turns out that large decoder-only models — when trained on enough data — develop strong understanding capabilities on their own, without needing a separate encoder. The original encoder-decoder design made more sense when models were smaller and the two components provided a clean separation of "understand the input" and "generate the output" — but at scale, that separation is no longer necessary.

## Summary

### Tokenization

| Concept | What It Does |
|---|---|
| **Token** | Basic unit of text (character, word, or subword) |
| **Vocabulary** | Dictionary mapping tokens to integer IDs |
| **BPE** | Learns subword splits from corpus; handles unseen words |
| **Sliding Window** | Turns a long sequence into input-target pairs for training |

### Attention Mechanisms

| Mechanism | Weights Trainable? | Key Feature |
|---|---|---|
| **Simple self-attention** | No | Dot product of raw input vectors |
| **Q, K, V self-attention** | Yes (Wq, Wk, Wv) | Separate learned projections for querying, matching, passing |
| **Causal attention** | Yes | Upper triangle masked; tokens only see the past |
| **Multi-head attention** | Yes (per head + Wo) | Multiple parallel heads; each learns different relationships |

### Architectures

| Architecture | Attention | Use Case | Examples |
|---|---|---|---|
| **Encoder-only** | Bidirectional | Understanding | BERT, RoBERTa |
| **Decoder-only** | Causal | Generation | GPT, LLaMA |
| **Encoder-Decoder** | Both + cross-attention | Seq-to-seq | T5, BART |

### Word Embeddings

| Method | Input | Predicts | Strength |
|---|---|---|---|
| **Skip-gram** | Center word | Context words | Rare words |
| **CBOW** | Context words | Center word | Frequent words, speed |
